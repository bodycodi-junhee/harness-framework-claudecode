## 문제해결 5단계 접근법

새 기능 구현 또는 기존 코드 수정은 항상 아래 순서로 진행한다. 문제를 받자마자 바로 코드로 들어가지 않는다.

### 1. Explore (탐색)
요구사항과 제약을 확인한다.

- 입력(Input)은 무엇인가? 어디서 오는가?
- 출력(Output)은 무엇이고, 어떤 형식인가? (출력이 음수가 되는 경우는 없는가?)
- 예외 상황은 무엇인가?
- 변경 가능성이 높은 부분(정책, 요금 체계, 할인 조건 등)은 어디인가?
- 최소 구현 범위(MVP)는 무엇인가?

### 2. Brainstorm (설계 비교)
설계 옵션을 1가지만 내지 않는다. 여러 옵션을 trade-off와 함께 비교한다.

- Option A: 장점 / 단점
- Option B: 장점 / 단점
→ 현재 요구사항에 맞는 합리적인 옵션을 선택하고 이유를 설명한다.

### 3. Plan (설계 확정)
선택한 설계를 구조화한다.

- 핵심 클래스/인터페이스를 나누고 각각의 책임을 정의한다
- 변경 가능성이 높은 부분(정책, 규칙)은 인터페이스/전략 패턴으로 격리한다
- 책임을 분리한다 — 서브에이전트가 parent context와 분리되어 작은 작업을 수행하듯, 좋은 설계도 큰 책임을 작은 단위로 나눈다

**Plan-First 발동 신호**: 아래 3가지 중 하나라도 해당하면 먼저 계획만 작성하고 사용자 확인 후 구현한다.
1. 영향 범위를 확신할 수 없을 때
2. 여러 디렉토리/서비스/모듈이 얽혀 있을 때
3. 레거시 코드라 수정 한 번이 연쇄 영향을 만들 수 있을 때

### 4. Implement (구현)
Explore → Brainstorm → Plan에서 확정된 설계대로 구현한다.

- 테스트 가능한 구조로 작성한다 (의존성 주입, 인터페이스 분리)
- 이 step에 명시된 범위만 수정한다. 추가 기능 금지.
- 기존 테스트를 깨뜨리지 않는다

### 5. Verify (검증)
"코드가 돌아간다"가 아니라 "요구사항을 만족하는지"를 증명한다.

```bash
./gradlew build   # 컴파일 에러 없음
./gradlew test    # 테스트 통과
# Maven: ./mvnw verify
```

## Harness 타입

Claude Code의 핵심 루프는 모델 판단 → Tool Use 요청 → Harness가 검증/실행 → 결과 반환의 반복이다.
step 설계 시 아래 6가지 Harness를 의식적으로 채워라.

| Harness          | 역할                      | step에서 해야 할 일                             |
| ---------------- | ------------------------- | ----------------------------------------------- |
| **Context**      | 무엇을 봐야 하는지        | 입력/출력/제약/변경가능성 4요소 명시            |
| **Permission**   | 무엇이 허용되는지         | 수정 가능 범위, 수정 금지 범위, Deny First 원칙 |
| **Tool**         | 어떤 도구를 어떻게 쓰는지 | 빌드/테스트 커맨드 명시                         |
| **Verification** | 결과를 어떻게 확인하는지  | AC 커맨드 + 5레이어 검증                        |
| **Subagent**     | 작업을 어떻게 분리하는지  | 조사/수정/검토 역할 분리                        |
| **Persistence**  | 상태를 어떻게 유지하는지  | index.json 업데이트, summary 기록               |

### Context Harness — 4요소

step을 작성할 때 아래 4요소를 명시한다. 이 중 하나라도 흐리면 Claude가 추측으로 움직인다.

1. **Input**: 입력값의 타입, 출처, 허용 범위
2. **Output**: 출력 형식, 허용되지 않는 값 (예: 음수 불가)
3. **제약**: 깨지면 안 되는 불변 조건 (예: 취소 주문 제외, 음수 거리 금지)
4. **변경가능성**: 정책적으로 바뀔 수 있는 부분 (예: 도시별 요금, 할인 기준)

### Deny First 원칙 (Permission Harness)

Claude Code가 모르는 액션은 그냥 허용하지 않고 막거나 사용자에게 확인하는 것처럼, 설계에서도 같은 원칙을 적용한다.

- 모르는 요구사항은 그냥 가정하지 않는다
- 위험한 입력(음수 금액, null 필수값, 허용되지 않은 상태 값)은 허용하지 않는다
- 애매한 정책은 코드 안에 숨기지 않는다 — 명시적으로 드러낸다

step 설계 시 반드시 포함할 것:
- **수정 가능 범위**: 건드려도 되는 파일/패키지/클래스
- **수정 금지 범위**: 절대 건드리면 안 되는 것 + **이유**
- **유지해야 할 조건**: 깨지면 안 되는 불변 조건

### Verification 5레이어

단일 테스트에만 의존하지 않는다. 아래 레이어를 순서대로 거쳐 안전장치를 겹쳐 쌓는다.

1. **타입 제한**: 잘못된 타입이 시스템에 들어오지 못하도록
2. **입력 검증**: 음수 금지, 필수값 누락 금지, 허용 범위 초과 금지
3. **정책 검증**: 비즈니스 규칙 (무료 배송 기준, 할인 적용 순서, 요금 조건)
4. **결과 검증**: 출력값이 허용 범위 안에 있는지 (예: 최종 금액 음수 불가)
5. **테스트**: Happy path + Edge case + Invalid input

### Sub-agent 역할 분리 (Subagent Harness)

서브에이전트는 자동화 장식이 아니라 작업 경계를 분리해서 메인 컨텍스트를 깨끗하게 유지하는 구조다.
서브에이전트는 자기 작업 전체를 parent에 넘기지 않고 **요약만 반환**한다.

- **조사용**: 원인 분석, 파일 탐색, 영향 범위 파악 (read-only)
- **수정용**: 제한된 범위만 수정
- **검토용**: diff 리뷰, 테스트 결과 확인

규칙: 역할이 겹치면 만들지 않는다. 역할이 분명할 때만 만든다.
작업 경계가 모호한 상태에서 서브에이전트부터 여러 개 만들면 오히려 혼란만 커진다.

## 세션 관리

세션이 오염되면 아무리 좋은 지시도 희석된다. 세션 품질을 운에 맡기지 않는다.

| 명령       | 언제 쓰는가                                                               | 주의                                                                           |
| ---------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `/clear`   | 작업 단위가 바뀔 때. 이전 작업 맥락이 다음 작업에 섞이지 않도록 완전 리셋 | 모든 컨텍스트 사라짐                                                           |
| `/compact` | 긴 작업을 계속해야 할 때. 대화를 요약해서 유지                            | **무엇을 남길지 명시 필수**: 변경 파일 목록, 실행 커맨드, 결과 요약, 남은 투두 |

**핵심 원칙**: 대화 속 규칙은 휘발성, 파일 속 규칙은 지속성.
`/compact` 이후에도 살아있어야 하는 결정·금지사항은 반드시 `CLAUDE.md` 또는 `docs/`에 승격시켜라.

## 워크플로우

### 0. 요구사항 명확화 (조건부 발동)

아래 중 하나라도 해당하면 탐색 전에 `/deep-interview`를 먼저 실행한다:

- 목표가 흐릿하다 ("이거 개선해줘", "더 좋게 만들어줘" 수준)
- 범위를 확신할 수 없다 (이 파일만? 전체 모듈? 다른 서비스도 포함?)
- 완료 기준이 없다 (언제 끝난 건지 판단할 수 없다)
- 여러 도메인/서비스에 걸쳐 있어 어디서 시작할지 모호하다

아래에 해당하면 이 단계를 건너뛴다:

- 단순 오타 수정, 작은 설정 변경, 테스트 보강처럼 요청 자체가 명확하다
- 파일명·함수명·조건처럼 구체적인 지시가 이미 포함되어 있다

**스킬 사전 확인**: `/deep-interview`를 실행하기 전에 스킬이 설치되어 있는지 확인한다.

```bash
# 스킬 존재 여부 확인
ls ~/.claude/skills/deep-interview/SKILL.md 2>/dev/null || \
  npx skills@latest add devbrother2024/skills --skill deep-interview --agent claude-code --global --yes
```

설치 확인 후 `/deep-interview`를 실행한다. 종료되면 정리된 요구사항(목표, 범위, 제약, 완료 기준)을 기반으로 A 단계를 시작한다.

### A. 탐색

`/docs/` 하위 문서(PRD, ARCHITECTURE, ADR)를 읽고 프로젝트의 기획·아키텍처·설계 의도를 파악한다. 필요시 Explore 에이전트를 병렬로 사용한다.

### B. 논의

구현을 위해 구체화하거나 기술적으로 결정해야 할 사항이 있으면 사용자에게 제시하고 논의한다.

### C. Step 설계

사용자가 구현 계획 작성을 지시하면 여러 step으로 나뉜 초안을 작성해 피드백을 요청한다.

설계 원칙:

1. **Scope 최소화** — 하나의 step에서 하나의 레이어 또는 모듈만 다룬다. 여러 모듈을 동시에 수정해야 하면 step을 쪼갠다.
2. **자기완결성** — 각 step 파일은 독립된 Claude 세션에서 실행된다. "이전 대화에서 논의한 바와 같이" 같은 외부 참조는 금지한다. 필요한 정보는 전부 파일 안에 적는다.
3. **사전 준비 강제** — 관련 문서 경로와 이전 step에서 생성/수정된 파일 경로를 명시한다. 세션이 코드를 읽고 맥락을 파악한 뒤 작업하도록 유도한다.
4. **시그니처 수준 지시** — 함수/클래스의 인터페이스만 제시하고 내부 구현은 에이전트 재량에 맡긴다. 단, 설계 의도에서 벗어나면 안 되는 핵심 규칙(Deny First, 멱등성, 보안, 데이터 무결성 등)은 반드시 명시한다.
5. **AC는 실행 가능한 커맨드** — `./gradlew build && ./gradlew test` 형태의 실제 실행 가능한 검증 커맨드를 포함한다.
6. **주의사항은 구체적으로** — "조심해라" 대신 "X를 하지 마라. 이유: Y" 형식으로 적는다.
7. **네이밍** — step name은 kebab-case slug (예: `domain-model`, `service-layer`, `api-controller`).

### D. 파일 생성

사용자가 승인하면 아래 파일들을 생성한다.

#### D-1. `phases/index.json` (전체 현황)

여러 task를 관리하는 top-level 인덱스. 이미 존재하면 `phases` 배열에 새 항목을 추가한다.

```json
{
  "phases": [
    {
      "dir": "0-mvp",
      "status": "pending"
    }
  ]
}
```

- `dir`: task 디렉토리명.
- `status`: `"pending"` | `"completed"` | `"error"` | `"blocked"`. execute.py가 실행 중 자동으로 업데이트한다.
- 타임스탬프(`completed_at`, `failed_at`, `blocked_at`)는 execute.py가 상태 변경 시 자동 기록한다. 생성 시 넣지 않는다.

#### D-2. `phases/{task-name}/index.json` (task 상세)

```json
{
  "project": "<프로젝트명>",
  "phase": "<task-name>",
  "steps": [
    { "step": 0, "name": "domain-model", "status": "pending" },
    { "step": 1, "name": "service-layer", "status": "pending" },
    { "step": 2, "name": "api-controller", "status": "pending" }
  ]
}
```

필드 규칙:

- `project`: 프로젝트명 (CLAUDE.md 참조).
- `phase`: task 이름. 디렉토리명과 일치시킨다.
- `steps[].step`: 0부터 시작하는 순번.
- `steps[].name`: kebab-case slug.
- `steps[].status`: 초기값은 모두 `"pending"`.

상태 전이와 자동 기록 필드:

| 전이          | 기록되는 필드                  | 기록 주체                                     |
| ------------- | ------------------------------ | --------------------------------------------- |
| → `completed` | `completed_at`, `summary`      | Claude 세션 (summary), execute.py (timestamp) |
| → `error`     | `failed_at`, `error_message`   | Claude 세션 (message), execute.py (timestamp) |
| → `blocked`   | `blocked_at`, `blocked_reason` | Claude 세션 (reason), execute.py (timestamp)  |

`summary`는 step 완료 시 산출물을 한 줄로 요약한 것으로, execute.py가 다음 step 프롬프트에 컨텍스트로 누적 전달한다. 따라서 다음 step에 유용한 정보(생성된 파일, 핵심 결정 등)를 담아야 한다.

`created_at`은 execute.py가 최초 실행 시 task 레벨에 한 번만 기록한다. step 레벨의 `started_at`도 execute.py가 각 step 시작 시 자동 기록한다. 생성 시 넣지 않는다.

#### D-3. `phases/{task-name}/step{N}.md` (각 step마다 1개)

```markdown
# Step {N}: {이름}

## 읽어야 할 파일

먼저 아래 파일들을 읽고 프로젝트의 아키텍처와 설계 의도를 파악하라:

- `/docs/ARCHITECTURE.md`
- `/docs/ADR.md`
- {이전 step에서 생성/수정된 파일 경로}

이전 step에서 만들어진 코드를 꼼꼼히 읽고, 설계 의도를 이해한 뒤 작업하라.

## Context Harness

- **Input**: {입력값, 타입, 허용 범위}
- **Output**: {출력 형식, 허용되지 않는 값}
- **제약**: {깨지면 안 되는 불변 조건}
- **변경가능성**: {정책적으로 바뀔 수 있는 부분}

## 작업

{구체적인 구현 지시. 파일 경로, 클래스/함수 시그니처, 로직 설명을 포함.
코드 스니펫은 인터페이스/시그니처 수준만 제시하고, 구현체는 에이전트에게 맡겨라.
단, 설계 의도에서 벗어나면 안 되는 핵심 규칙은 명확히 박아넣어라.}

## Acceptance Criteria

```bash
./gradlew build   # 컴파일 에러 없음
./gradlew test    # 테스트 통과
# Maven: ./mvnw verify
```

## 검증 절차

1. 위 AC 커맨드를 실행한다.
2. Verification 5레이어를 확인한다:
   - 타입 제한이 적용되었는가?
   - 입력 검증 (음수, null, 허용 범위)이 있는가?
   - 정책 검증이 명시적으로 드러나 있는가?
   - 결과값이 허용 범위 안에 있는가?
   - Happy path + Edge case 테스트가 있는가?
3. 아키텍처 체크리스트를 확인한다:
   - ARCHITECTURE.md 레이어 의존성 규칙을 따르는가?
   - ADR 기술 스택을 벗어나지 않았는가?
   - CLAUDE.md CRITICAL 규칙을 위반하지 않았는가?
4. 결과에 따라 `phases/{task-name}/index.json`의 해당 step을 업데이트한다:
   - 성공 → `"status": "completed"`, `"summary": "산출물 한 줄 요약"`
   - 수정 3회 시도 후에도 실패 → `"status": "error"`, `"error_message": "구체적 에러 내용"`
   - 사용자 개입 필요 (API 키, 외부 인증, 수동 설정 등) → `"status": "blocked"`, `"blocked_reason": "구체적 사유"` 후 즉시 중단

## 수정 가능 범위

### E. 실행

```bash
python3 scripts/execute.py {task-name}        # 순차 실행
python3 scripts/execute.py {task-name} --push  # 실행 후 push
```

execute.py가 자동으로 처리하는 것:

- `feat-{task-name}` 브랜치 생성/checkout
- 가드레일 주입 — CLAUDE.md + docs/*.md 내용을 매 step 프롬프트에 포함
- 컨텍스트 누적 — 완료된 step의 summary를 다음 step 프롬프트에 전달
- 자가 교정 — 실패 시 최대 3회 재시도하며, 이전 에러 메시지를 프롬프트에 피드백
- 2단계 커밋 — 코드 변경(`feat`)과 메타데이터(`chore`)를 분리 커밋
- 타임스탬프 — started_at, completed_at, failed_at, blocked_at 자동 기록
- `HARNESS_AUTOMATED=1` 환경변수 주입 — Stop hook 중복 실행 방지

에러 복구:

- **error 발생 시**: `phases/{task-name}/index.json`에서 해당 step의 `status`를 `"pending"`으로 바꾸고 `error_message`를 삭제한 뒤 재실행한다.
- **blocked 발생 시**: `blocked_reason`에 적힌 사유를 해결한 뒤, `status`를 `"pending"`으로 바꾸고 `blocked_reason`을 삭제한 뒤 재실행한다.
