이 프로젝트의 변경 사항을 리뷰하라.

먼저 다음 문서들을 읽어라:
- `/CLAUDE.md`
- `/docs/ARCHITECTURE.md`
- `/docs/ADR.md`

그런 다음 변경된 파일들을 확인하고, 아래 체크리스트로 검증하라:

## 체크리스트

1. **아키텍처 준수**: ARCHITECTURE.md에 정의된 디렉토리 구조와 레이어 의존성 규칙(Presentation → Application → Domain; Infrastructure → Domain)을 따르고 있는가?
2. **기술 스택 준수**: ADR에 정의된 기술 선택을 벗어나지 않았는가?
3. **수정 범위**: 수정 범위가 요구사항에 맞는가? 건드리면 안 되는 파일(`.env`, GitHub workflows, public API 인터페이스 등)을 건드리지 않았는가?
4. **Deny First 준수**: 알 수 없는 요구사항을 임의로 가정하지 않았는가? 위험한 입력이나 동작을 허용하지 않았는가? 모호한 정책을 코드 안에 숨기지 않았는가?
5. **CRITICAL 규칙**: CLAUDE.md의 CRITICAL 규칙을 위반하지 않았는가?
6. **검증 5개 레이어**:
   - 타입 제한: 허용 타입이 올바르게 제한되었는가?
   - 입력값 검증: 음수·필수 필드·경계값 검증이 있는가?
   - 정책 검증: 비즈니스 정책(할인, 수수료, 조건 등)이 올바르게 적용되었는가?
   - 결과 검증: 최종 결과가 제약 조건(음수 금지 등)을 만족하는가?
   - 테스트: happy path + edge case + invalid input 케이스가 모두 작성되었는가?
7. **테스트 존재**: 새로운 기능에 대한 단위 테스트와 통합 테스트가 작성되어 있는가?
8. **검증 비어있음 금지**: "완료"라고 했는데 실제 검증(테스트 실행, 빌드 확인)이 빠져 있지는 않은가?
9. **빌드 가능**: `./gradlew build && ./gradlew test` (또는 `./mvnw test`) 명령이 에러 없이 통과하는가?
10. **민감정보 노출 금지**: 비밀키, API Key, DB 접속정보, 개인정보(PII)가 코드·로그에 포함되지 않았는가?

## Skills vs Tools 판단

리뷰 중 반복 패턴이 보이면 다음 기준으로 Skills/Tools 도입 여부를 판단하라:

| 상황 | 판단 |
|------|------|
| 같은 절차를 3회 이상 반복 설명 | Skills 후보 |
| 외부 시스템 호출 필요 | Tools/MCP 후보 |
| 절차 표준화가 필요하지만 외부 연결 불필요 | Skills |
| Claude가 절차는 알지만 실행 능력이 없음 | Tools |
